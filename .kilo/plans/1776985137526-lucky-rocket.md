# Plan: Implement Search API for Facility Detection in FeatureDialog

## Problem Analysis
The current implementation in FeatureDialog.tsx only detects visible schools on the current map tile because it uses `map.queryRenderedFeatures()` which only returns features that are currently rendered in the viewport. To detect all schools regardless of visibility, we need to either:
1. Query the source directly (which was partially implemented in the recent fix)
2. Use a search API that can query all facilities regardless of viewport

The user is asking about using a search API approach instead.

## Proposed Solution
Create a search API endpoint that can find nearby facilities (schools, colleges) and hazards, then modify FeatureDialog to use this API instead of direct map queries.

### Steps:
1. Create API routes for facility search:
   - `/app/api/facilities/route.ts` - Search for schools/colleges
   - `/app/api/hazards/route.ts` - Search for hazard data

2. Modify FeatureDialog.tsx:
   - Replace `scanForFacilities` to call the facilities API
   - Replace `scanForHazards` to call the hazards API (where applicable)
   - Remove direct map querying logic

3. Ensure the APIs can access the necessary data:
   - Schools data likely comes from GeoJSON files in `/data/geojson/`
   - Hazard data may come from various sources including GeoTIFF (for landslide) and GeoJSON (for fault lines, flood zones)

### API Design:
**Facilities API** (`/app/api/facilities/route.ts`):
- Parameters: `lng`, `lat`, `type` (elementary/secondary/higher), `radius` (optional)
- Returns: Nearest facilities of specified type within radius

**Hazards API** (`/app/api/hazards/route.ts`):
- Parameters: `lng`, `lat`
- Returns: Distance to nearest fault line, flood zone, and landslide intensity

### Implementation Details:
1. The facilities API would read from schools GeoJSON data and calculate distances
2. The hazards API would:
   - For landslide: Use existing `/api/landslide` endpoint
   - For fault/flood: Query relevant GeoJSON files and calculate distances
3. Update FeatureDialog to use these APIs with proper loading/error states

### Benefits:
- No longer dependent on map viewport for facility detection
- Consistent performance regardless of zoom level
- Potential for better caching and optimization
- Separation of concerns (data fetching vs UI)

### Challenges:
- Need to implement efficient spatial search algorithms
- May need to load large GeoJSON files into memory for API routes
- Need to ensure API performance is acceptable

## Next Steps:

2. Create API routes
3. Update FeatureDialog to use APIs
