# FeatureDialog Refactoring Plan

## Executive Summary

Transform the monolithic 1,457-line `FeatureDialog.tsx` into a modular, maintainable component architecture. Separate concerns into distinct layers: **state/hooks**, **business logic**, **UI components**, and **composition**.

---

## Current State Analysis

### File Stats
- **Total**: ~1,457 lines
- **Core Responsibilities**:
  1. Mapbox reverse geocoding & address parsing
  2. Hazard scanning (faults, flood, landslide, volcanoes)
  3. Facility finding (schools, education)
  4. Income data lookup (province/region/ national)
  5. Directions routing (walking/driving)
  6. 4 tab UIs (Overview, Hazards, Economy, Education)
  7. State management for all above
  8. Utility components (MiniBarChart, Sparkline, RiskBar, RadialGauge)

### Problems Identified
1. **Single Responsibility Violation**: One component handles UI, data fetching, business logic, state
2. **Unreusable Logic**: `findIncomeData`, hazard scanning, facility finding are tightly coupled
3. **Difficult Testing**: Hard to unit test isolated pieces
4. **Performance Issues**: All effects and state live in one component, causing unnecessary re-renders
5. **Maintenance Burden**: 1.4k lines makes changes risky and slow
6. **Props Sprawl**: Props interface is minimal, but internal complexity is hidden

## Component Breakdown

### 1. `FeatureDialog.tsx` (Orchestrator)

**Responsibilities**:
- Hold top-level state (activeTab, locationAddress, incomeDataState, hazard states, facility states)
- Initialize and coordinate hooks
- Render dialog wrapper and tab navigation
- Pass data/actions to tab components
- Handle dialog lifecycle (open/close, escape key)

**Props**:
```typescript
interface FeatureDialogProps {
  isOpen: boolean;
  onClose: () => void;
  properties: FeatureProperties;
  titleKey?: string;
  descriptionKeys?: string[];
  position?: { lng: number; lat: number };
  mapRef?: React.RefObject<mapboxgl.Map | null>;
  onShowRoute?: (geometry: GeoJSON.LineString, profile: "walking" | "driving") => void;
}
```

**State** (lifted from 600+ lines):
```typescript
const [activeTab, setActiveTab] = useState<DialogTab>("overview");
const [locationAddress, setLocationAddress] = useState<string | null>(null);
const [incomeDataState, setIncomeDataState] = useState<IncomeData | null>(null);
const [hazardData, setHazardData] = useState<HazardDataState>({...});
const [facilityData, setFacilityData] = useState<FacilityDataState>({...});
const [isScanning, setIsScanning] = useState({ hazards: false, facilities: false });
```

**Size**: ~250 lines total

### 2. Hook Modules

#### `hooks/useReverseGeocode.ts`
- Purpose: Call Mapbox reverse geocoding API
- Input: `position` coordinates
- Output: `address` string
- Logic: Fetch from Mapbox, set `locationAddress`, error handling

#### `hooks/useIncomeLookup.ts`
- Purpose: Hierarchical income data resolution
- Input: `locationName`, `address`
- Output: `IncomeData` (normalized)
- Logic: Parse address тЖТ search city тЖТ province тЖТ region тЖТ national
- Exported helper: `findIncomeData()` (already existing logic, needs cleanup)

#### `hooks/useHazardScan.ts`
- Purpose: Scan map layers for hazards near position
- Input: `mapRef`, `position`
- Output: `HazardData` (faultDistance, floodDistance, landslideIntensity, volcanoCount, activeVolcanoes)
- Logic: Query map source features, call `findNearestHazardLine`, generate dummy volcano data
- Note: Keep temporary random generation but isolate it

#### `hooks/useFacilityScan.ts`
- Purpose: Find nearest schools (elementary + higher ed)
- Input: `mapRef`, `position`
- Output: `FacilityData` (nearestElementary, nearestHigherEd, directions)
- Logic: Query `Schools` layer with maki filters, call `findNearestFacility`, fetch directions

**Pattern**: Each hook returns `{ data, isScanning, error, refetch }` for testability and reusability.

## Recommended Approach: Top-Down Composition

**Make FeatureDialog a thin orchestrator**:
```tsx
export default function FeatureDialog(props) {
  const state = useFeatureDialogState(props); // custom hook
  return (
    <Dialog>
      <TabNavigation activeTab={state.activeTab} onTabChange={state.setActiveTab} />
      {state.activeTab === "overview" && <OverviewTab {...state} />}
      {state.activeTab === "hazards" && <HazardsTab {...state} />}
      ...
    </Dialog>
  );
}
```

**Why this pattern?**
- Easy to follow execution flow
- State clearly passed down
- Each tab independently testable
- Simple to add new tabs
- No context boilerplate
